## Summary

Protect latest-version semantics from silent corruption when multiple writers target the same lineage.

## Goal

Implement `IfLatestID`-based optimistic locking and conflict detection for contested update flows.

## Scope

In scope:

- `IfLatestID` checks on update writes
- conflict rejection for stale latest assumptions
- preservation of single-latest lineage state under contention

Out of scope:

- generic distributed locking
- migration locking
- derived-specific conflict policies

## Deliverables

- optimistic lock enforcement in contested write path
- conflict error behavior for stale updates

## Outcome

- `Remember` now enforces `IfLatestID` against the current lineage latest inside the write transaction.
- Stale writers now receive explicit `ErrConflict` responses.
- Conflict tests verify that rejected writes do not mutate `latest/` or live corpus state.

## Acceptance Criteria

1. Contested updates using `IfLatestID` reject stale writers instead of corrupting lineage state.
2. Successful updates still preserve a single latest record for the lineage.
3. Conflict detection integrates with the existing write path rather than bypassing it.
