## Summary

Prove that the new advisory curation APIs remain explainable cold-path diagnostics and do not alter hot-path retrieval, lineage behavior, or governed promotion semantics.

## Goal

Build a verification layer that demonstrates `SynthesisCandidates` and `PromotionSuggestions` improve curator visibility without creating semantic drift in retrieval or write behavior.

## Scope

In scope:

- verification of advisory-only behavior
- verification of live-only default scope
- verification of forgotten/history opt-in behavior
- regression checks for `Recall` and `GetProfile`
- regression checks for relation and latest-state non-mutation
- false-positive and noisy-corpus cases

Out of scope:

- benchmarking model quality improvements
- UI acceptance testing for curator workflows
- write-flow tests unrelated to advisory API boundaries

## Deliverables

- verification tests for advisory-only API behavior
- regression coverage proving no `Recall` or `GetProfile` changes
- regression coverage proving no relation or latest-state mutation
- noisy and contradictory fixture coverage
- verification notes explaining why the APIs remain cold-path diagnostics

## Acceptance Criteria

1. Tests prove that calling `SynthesisCandidates` cannot change `Recall` results.
2. Tests prove that calling `PromotionSuggestions` cannot change `GetProfile` results.
3. Tests prove that neither API creates relations, write records, audit mutations, or latest-state transitions.
4. Tests prove forgotten and historical memories are excluded by default and included only by explicit request.
5. Tests prove similarity clusters do not imply duplicate truth or relation truth.
6. Tests prove recurrent episodic candidates do not bypass governed promotion behavior.
7. Tests cover noisy, churny, or contradictory cases and verify downgrade behavior rather than overconfident suggestion output.

## Notes

This ticket is the semantic guardrail. The key question is not only whether the advisory APIs return plausible candidates, but whether they remain fully separate from hot-path retrieval, explicit lineage, and governed durable-truth creation.

Verification progress so far:

- store tests prove live-only defaults for advisory curation APIs
- store tests prove explicit historical and forgotten opt-in behavior
- store tests prove `SynthesisCandidates` does not change `Recall`
- store tests prove `SynthesisCandidates` and `PromotionSuggestions` do not change `GetProfile`
- store tests prove advisory curation calls do not mutate audit log state
- store tests prove advisory curation calls do not mutate relation traversal or lineage latest-state behavior
- store tests cover downgrade behavior for blob-like synthesis clusters and churny or contradiction-like promotion support clusters

Residual follow-up that may still be worth adding later:

- broader false-positive fixtures beyond the current focused store scenarios
- optional end-to-end verification notes tying HTTP, CLI, and MCP surfaces back to the same advisory-only contract
