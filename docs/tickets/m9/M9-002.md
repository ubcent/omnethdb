## Summary

Implement cold-path advisory APIs that surface synthesis-review clusters and promotion-review suggestions for memory curators without changing retrieval, lineage, or governed write behavior.

## Goal

Build `SynthesisCandidates` and `PromotionSuggestions` on top of the existing live corpus, similarity tooling, and inspector-facing quality surfaces so curators can inspect higher-order curation opportunities explicitly.

## Scope

In scope:

- implementation of `SynthesisCandidates`
- implementation of `PromotionSuggestions`
- shared reason-code and evidence plumbing
- default live-only filtering
- inspector-facing integration or internal service surface for later exposure
- tests for advisory output shape and scope boundaries

Out of scope:

- automatic synthesis
- automatic promotion
- automatic relation creation
- changes to `Recall` or `GetProfile`
- write-path mutations triggered by advisory outputs
- human workflow UI beyond basic exposure of returned diagnostics

- tests for live-only defaults, evidence fields, and advisory-only behavior
- integration notes for inspector, CLI, HTTP, or MCP exposure

## Acceptance Criteria

1. `SynthesisCandidates` returns clusters of live latest episodic memories with stable reason codes and supporting evidence.
2. `PromotionSuggestions` returns single episodic memory or lineage review candidates with stable reason codes and supporting evidence.
3. Both APIs exclude forgotten and historical memories by default and include them only when explicitly requested.
4. Both APIs produce advisory output only and do not create writes, relations, or latest-state changes.
5. No implementation path changes the behavior of `Recall` or `GetProfile`.
6. Churny, contradictory, or blob-like candidates are downgraded rather than surfaced as strong promotion or synthesis suggestions.
7. The implementation is structured so inspector, CLI, HTTP, and MCP surfaces can reuse one behavior model.

## Notes

Implementation should bias toward explicit evidence over opaque ranking. `SynthesisCandidates` and `PromotionSuggestions` must remain separate internal concepts even if they share utilities for live-corpus filtering, similarity signals, and reason-code generation.

Implemented through:

- public request and result types for advisory curation APIs
- bolt-store implementations for `GetSynthesisCandidates` and `GetPromotionSuggestions`
- default live-only behavior with explicit historical and forgotten opt-in
- HTTP endpoints, CLI commands, inspector integration, MCP tools, and gRPC RPCs over the same behavior model
- focused tests for advisory output shape, scope boundaries, and downgrade behavior
