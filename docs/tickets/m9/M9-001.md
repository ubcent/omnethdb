## Summary

Define two explicit cold-path advisory APIs that help memory curators identify clusters of episodic memories worth synthesis review and individual episodic memories worth promotion review, without changing retrieval or write semantics.

## Goal

Produce a stable contract for `SynthesisCandidates` and `PromotionSuggestions` that improves curation visibility while preserving explicit lineage, governed promotion, and predictable hot-path retrieval.

## Scope

In scope:

- `SynthesisCandidates` request and response shape
- `PromotionSuggestions` request and response shape
- reason-code and evidence vocabulary
- default live-only scope rules
- boundaries between advisory diagnostics and write semantics
- acceptance-oriented examples for ambiguous and noisy cases

Out of scope:

- automatic synthesis
- automatic promotion
- automatic relation creation
- retrieval ranking changes
- new write-path side effects
- hidden governance bypasses

## Deliverables

- advisory curation API contract doc or code-level type definition
- explicit reason-code taxonomy for synthesis and promotion review
- examples showing separation between cluster synthesis review and single-memory promotion review
- documented invariants forbidding retrieval and write side effects

## Acceptance Criteria

1. The contract defines `SynthesisCandidates` and `PromotionSuggestions` as separate APIs with distinct review semantics.
2. Both APIs default to live latest memories only, with forgotten or historical data included only by explicit request.
3. The contract states that similarity, recurrence, and cumulative score can surface review candidates but cannot imply lineage, truth, or promotion by themselves.
4. The contract states that neither API can affect `Recall`, `GetProfile`, relation state, or latest-state transitions.
5. `SynthesisCandidates` is defined as cluster-level advisory output for curator review rather than duplicate truth or relation recommendation.
6. `PromotionSuggestions` is defined as single-memory or lineage-level advisory output for governed promotion review rather than promotion execution.
7. The contract includes stable reason codes and visible supporting evidence for why each candidate surfaced.
8. The contract is specific enough for inspector, CLI, HTTP, and MCP surfaces to share one behavior model.

## Notes

The main design constraint is that these APIs must improve curator judgment without becoming hidden product logic. `SynthesisCandidates` reviews clusters of episodic observations for possible synthesis. `PromotionSuggestions` reviews a single episodic memory or lineage for possible governed promotion. The contract should prefer explainable reason codes over opaque scoring and should explicitly downgrade churny, contradictory, or blob-like cases to `review_only`.
