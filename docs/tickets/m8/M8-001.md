## Summary

Define an explicit advisory lint contract for candidate memory writes so quality signals can improve write discipline without changing memory semantics.

## Goal

Produce a stable contract for pre-write or inline write linting that can identify likely duplicates, likely update targets, and suspicious blob-like memories.

## Scope

In scope:

- lint request and response shape
- warning and suggestion codes
- rules for similarity-assisted duplicate and update hints
- boundaries between advisory signals and enforced policy

Out of scope:

- automatic relation creation
- automatic blocking of writes
- auto-splitting or rewriting candidate memories

## Deliverables

- write-lint contract doc or code-level type definition
- explicit warning taxonomy
- acceptance-oriented examples for duplicate and update suggestions

## Acceptance Criteria

1. The lint contract distinguishes advisory warnings from policy enforcement.
2. Similarity signals never imply lineage, truth, or promotion by themselves.
3. The contract includes at least `possible_duplicate`, `possible_update_target`, and `mixed_fact_blob` warnings.
4. The contract is specific enough for store, CLI, HTTP, and MCP integration to share one behavior model.
